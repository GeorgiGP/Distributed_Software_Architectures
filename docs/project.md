## 1. Въведение

Паралелните и разпределени системи са ключови за решаването на изчислително интензивни задачи в съвременния свят. Един от основните предизвикателства при такива системи е ефективното разпределение на натоварването между процесорите/възлите, особено когато работното натоварване е нехомогенно или непредвидимо.

В централизираните архитектури (Master-Slave) един главен процес координира разпределението на задачите. Този подход е прост за имплементация, но създава "тясно място" (bottleneck) при мащабиране. Алтернативата е **Peer-to-Peer (P2P)** архитектура, където процесите комуникират директно помежду си без централен координатор.

Целта на този проект е да се изследва архитектурата на P2P паралелни приложения при разпределено динамично балансиране, като се анализират различни топологии и подходи за паралелизация.

## 2. Теоретична основа

### 2.1. Паралелни модели и SPMD

При паралелното програмиране се различават два основни модела:

- **SPMD (Single Program Multiple Data)** - всички процеси изпълняват една и съща програма, но върху различни данни.
- **MPMD (Multiple Program Multiple Data)** - различните процеси могат да изпълняват различни програми.

### 2.2. Peer-to-Peer архитектури

При P2P архитектурите няма централен координатор - всички възли (peers) са равнопоставени.

**Предимства:**
- Няма единична точка на отказ
- По-добра скалируемост
- Разпределено вземане на решения

**Недостатъци:**
- По-сложна координация
- Възможно неоптимално разпределение
- Комуникационни разходи при пълен граф

### 2.3. Топологии на процесите

#### Верижна топология (Chain/Ring)
```
[P0] <---> [P1] <---> [P2] <---> [P3]
```
- Брой връзки: O(n)
- Комуникационна сложност: O(n) за глобална информация

#### Пълен граф (All-to-All)
```
    [P0]
   / | \ 
 [P1]-[P2]
   \ | /
    [P3]
```
- Брой връзки: O(n²)
- Комуникационна сложност: O(1) за директна комуникация

### 2.4. Динамично балансиране

**Work Stealing:**
1. Процес завършва задача
2. Ако опашката му е празна, поисква работа от съсед
3. Ако съседът има работа, споделя половината

## 4. Проектиране

### Реализирани варианти:

1. **Статично циклично балансиране** - базова линия
2. **P2P Work Stealing с пълен граф**
3. **P2P с декомпозиция на Фибоначи** (нов подход)

### Проблем: Грануларност при Фибоначи

При стандартния подход всяка задача `fib(n)` се изчислява изцяло от един процес:

```
Без декомпозиция:
P0: |====fib(45) - 2.5 секунди====|
P1: |=fib(35)=|----idle 1.8s------|
P2: |=fib(36)=|----idle 1.7s------|
P3: |=fib(37)=|----idle 1.6s------|
```

Това ограничава паралелизма - един процес е "bottleneck".

### Решение: Декомпозиция на задачите

Разбиваме `fib(n)` на подзадачи, които се разпределят:

```
fib(45) = fib(44) + fib(43)
        ↓           ↓
      P0          P1
       ↓           ↓
   fib(43)     fib(42)
   fib(42)     fib(41)
      ...        ...
```

Когато `n ≤ праг` (напр. 30) - изчисляваме директно.

## 5. Имплементация

### Файлове:
- `static_fine.c` - статично циклично балансиране
- `p2p_chain_fine.c` - P2P с верижна топология
- `p2p_full_fine.c` - P2P с пълен граф
- `p2p_fib_decomposition.c` - **P2P с декомпозиция**

### Ключови структури (декомпозиция):

```c
typedef struct {
    int n;            // Числото за изчисление
    int task_id;      // Уникален ID
    int parent_id;    // ID на родителя
    int parent_rank;  // Кой процес държи родителя
    int is_left;      // Ляв или десен наследник
} Task;

typedef struct {
    int task_id;
    long long left_result;
    long long right_result;
    int left_done, right_done;
    // ... parent info за комбиниране
} PartialResult;
```

## 9. Заключение

### Ключови изводи:

1. **Статичното балансиране** има таван ~1.5x ускорение при нехомогенно натоварване

2. **Work Stealing** подобрява до ~2.5x, но е ограничено от грануларността на задачите

3. **Декомпозицията** постига ~5.7x при 8 процеса (близко до линейно ускорение)

4. **Грануларността е критична** - задачи, които не могат да се разбият, ограничават паралелизма

5. **CPU балансът** е най-добрият индикатор за ефективност

### Препоръки за линейно ускорение:

1. Анализирай задачата - може ли да се декомпозира?
2. Избери подходяща грануларност (не прекалено фина, не прекалено едра)
3. Използвай work stealing за динамично балансиране
4. Мониторирай CPU натоварването за идентифициране на дисбаланс
